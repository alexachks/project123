# Event Vendors Marketplace (San Diego)

Маркетплейс вендоров для ивентов (свадьбы, дни рождения, похороны, корпоративы).
Клиент находит вендоров (Coffee Cart, Balloons, Catering и др.), общается с ними
и бронирует. Двусторонний маркетплейс: клиенты и вендоры — обе стороны с аккаунтами.

**Текущий этап:** фундамент. Фронт-сайт + БД. Платежей пока нет.

**Почему Supabase:** auth, storage, realtime и RLS из коробки. Фаундер — нетехнический,
Supabase даёт визуальный доступ к данным. Под ним обычный Postgres — миграция возможна.

## Ключевые архитектурные решения

### 1. Вендор — отдельная сущность, не роль пользователя

Таблица `vendors` со связью `owner_id → auth.users`. **Не** поле `role` на профиле.

Почему:
- Один человек может быть и клиентом, и вендором (владелица кейтеринга заказывает шарики на свою свадьбу)
- У вендора ~10 своих полей (бизнес, категория, галерея, зона выезда, Stripe-аккаунт), у клиента их нет → иначе таблица с половиной `NULL`
- Один бизнес = несколько сотрудников (позже: таблица `vendor_members`)

### 2. Город закладываем в схему сразу

Поле `city` / `service_area` есть с первого дня, но UI — только под Сан-Диего.
Расширение на LA потом не потребует миграции.

### 3. Цены считаются только на сервере

Итоговую сумму **никогда** не берём с клиента. Только пересчёт на сервере
(Route Handlers / Edge Functions). Иначе через DevTools можно оплатить свадьбу за $1.

### 4. Доступ через RLS, а не только в коде

Правила «вендор видит только свои букинги, клиент — только свои чаты» задаются
на уровне БД. Для маркетплейса с деньгами это критично.

### 5. Order ≠ Booking

Один заказ (`orders`) может содержать несколько бронирований (`bookings`) —
у каждого свой вендор, свой статус, своя выплата. Заложено в схему заранее,
даже если в MVP корзина будет с одним вендором.

### 6. Калькулятор: начинаем с шаблонов, не с конструктора

Вендор выбирает готовый тип ценообразования (per-person, per-hour, fixed+addons, tiered)
и заполняет цифры. Полный конструктор формул — только после того, как увидим
реальные прайсы 10+ вендоров. Причина: строить универсальный движок под неизвестные
требования — способ потратить 2 месяца впустую.

## Модель данных (черновик)

Финальная схема — после ответов на открытые вопросы.

```
auth.users              — Supabase Auth
profiles                — id(FK users), full_name, avatar_url, phone, created_at
vendors                 — id, owner_id(FK users), business_name, slug, category_id,
                          description, city, service_radius, status(draft|pending|
                          published|suspended), rating, created_at
vendor_media            — id, vendor_id, url, type, sort_order
categories              — id, slug, name, icon
event_types             — id, slug, name          (wedding, funeral, birthday, ...)

conversations           — id, client_id, vendor_id, created_at
messages                — id, conversation_id, sender_id, body, read_at, created_at

orders                  — id, client_id, event_date, event_type_id, status, total, created_at
bookings                — id, order_id, vendor_id, status, amount, details(jsonb)

pricing_templates       — id, vendor_id, type, config(jsonb)    -- фаза 2
```

Ключевое: `bookings.details` — jsonb с ответами квиза-калькулятора.
Схема ответов зависит от типа шаблона.

|---|---|---|

Схему БД проектируем сразу под фазу 3-4, UI выкатываем постепенно.
Тогда «включить оплаты» — не переписывание, а активация готового куска.

## Функциональность по состоянию авторизации

**Без регистрации:** каталог вендоров, страница вендора, фото, описание, категории, поиск/фильтры.

**Скрыто до регистрации:** контакты вендора, кнопка «написать», кнопка букинга.

**После регистрации:** аватар в хедере с меню (профиль, мои чаты, история букингов),
на странице вендора — кнопки букинга и сообщения.

## Открытые вопросы

Нужны ответы до старта соответствующих фаз.

### Блокируют финальную схему БД
1. **Корзина с несколькими вендорами** — нужна в первой версии или один вендор за заказ?
2. **Деньги при отказе вендора** — полная оплата с возвратом / депозит + остаток / подтверждение до списания?
   (Влияет на всю модель статусов `bookings`.)

### Блокируют фазу 2
3. **Модерация вендоров** — админ проверяет перед публикацией, или публикуются сразу?
4. **Несколько сотрудников у одного вендора** — нужно ли (таблица `vendor_members`)?

### Продуктовые
5. **Дедлайн / бюджет** — на что ориентироваться, что резать.
6. **Комиссия** — размер, кто платит (клиент/вендор/оба).

### Прочее
7. **Имя папки `project[123]`** — квадратные скобки ломают glob-паттерны в части
   инструментов. Переименовать сейчас дешевле, чем потом.

## Правила разработки

Общие правила — в `~/.claude/CLAUDE.md`. Специфичное для проекта:

- **Никаких секретов в клиентском коде.** `SUPABASE_SERVICE_ROLE_KEY` — только на сервере.
- **RLS включён на всех таблицах** с данными пользователей. Без исключений.
- **Каждая таблица** — с миграцией в `supabase/migrations/`, не через UI.
- **Цены и платежи** — только серверная логика.
- Бизнес-логика в `services/`, не в компонентах.
- Reusable-компоненты в `components/ui/`, специфичные — в `components/features/`.
